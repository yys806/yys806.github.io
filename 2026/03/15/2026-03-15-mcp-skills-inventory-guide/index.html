我的MCP与Skills清单说明

🧭 为什么要做这份清单

最近我把本机常用的 Skills 和 MCP 做了一次系统盘点。目的很简单：把“会用”升级成“稳定复用”。

这篇是我的公开版说明文档：写给未来的自己，也写给同样在搭 AI 工作流的人。

来源：skills.sh

🔧 Skills 全清单（18个，逐项说明）

下面是我当前环境里全部可用 skills，分为本地安装（14个）和内置（4个）。每一个都写“适合做什么”和“什么时候用最值”。

本地安装 skills（14个）

  1. RedBookSkills

    • 用途：自动发布小红书图文/视频，做登录检查、检索、评论互动、数据抓取。
    • 适用：内容运营标准化；减少重复手工发布与抄写。

  2. yt-dlp-downloader

    • 用途：下载多平台视频、提取音频、拉字幕、控制清晰度。
    • 适用：素材采集、离线备份、二次剪辑前的批量归档。

  3. blog

    • 用途：围绕 Hexo 的写作、改稿、排版、构建、发布流程。
    • 适用：持续写博客时，把“写-改-发”流程固定下来。

  4. website-organizer

    • 用途：维护 WebStack-Hugo 导航站，增改站点、封面/图标抓取、构建发布。
    • 适用：长期维护网址导航，避免条目格式不一致。

  5. xlsx

    • 用途：创建/修改表格、公式计算、格式整理、数据分析。
    • 适用：报表、对账、数据清洗和批量结构化处理。

  6. pdf

    • 用途：PDF 文本提取、拆分合并、表单填写、批处理转换。
    • 适用：资料归档、文档加工、统一输出交付件。

  7. docx

    • 用途：Word 文档编辑、批注、修订、格式保留与重排。
    • 适用：方案文档、评审文档、规范类文稿维护。

  8. pptx

    • 用途：生成/修改演示文稿，调整布局、提炼要点、补讲稿。
    • 适用：快速准备汇报、答辩、复盘型演示材料。

  9. paper-2-web

    • 用途：论文转网页、视频、海报的传播链路生成。
    • 适用：学术内容对外传播，统一多种展示形态。

  10. frontend-design

    • 用途：生成高完成度前端界面，强调视觉质量与可落地代码。
    • 适用：活动页、作品页、产品展示页快速落地。

  11. find-skills

    • 用途：根据任务目标检索可安装/可复用 skills。
    • 适用：先找现成能力再开工，降低重复造轮子。

  12. skill-creator

    • 用途：指导如何创建或升级 skill（结构、说明、触发条件）。
    • 适用：把高频任务沉淀成团队可复用能力。

  13. planning-with-files

    • 用途：复杂任务的文件化计划管理（计划、进度、发现记录）。
    • 适用：多阶段任务、长周期项目、多人接力协作。

  14. netlify-auto-deploy

    • 用途：站点部署到 Netlify，并固化自动发布流程。
    • 适用：静态站点发布上线，减少部署脚本手工操作。

内置 skills（4个）

  1. playwright

    • 用途：浏览器自动化、网页验证、截图、回归检查。
    • 适用：功能回归、自动化验收、线上页面巡检。

  2. frontend-ui-ux

    • 用途：强调 UI/UX 质量的前端设计与实现。
    • 适用：需要视觉一致性和体验细节的前端任务。

  3. git-master

    • 用途：规范 Git 流程（提交、历史追踪、分支整理、问题定位）。
    • 适用：改动多、协作多、需要干净历史时。

  4. dev-browser

    • 用途：持久化浏览器状态的自动化操作。
    • 适用：需要连续多步操作、会话保持、反复验证的网页任务。

什么时候优先用 Skill？

如果一个任务满足下面两条，就值得固化成 Skill：

  1. 重复频率高（每周都要做）
  2. 步骤固定（可标准化）

🧠 MCP 能力全清单（11类，逐项说明）

下面是我当前环境中可用的 MCP 能力分组。为了避免“只列名字没法用”，每组都写了核心能力和最适合的应用场景。

来源：modelcontextprotocol/docs

1) 开发协作：GitHub MCP

  • 能做什么：仓库检索、代码搜索、Issue/PR 增删改查、Review 评论、Merge、标签和里程碑管理、检查状态读取；还支持 Copilot 任务指派与状态追踪。
  • 什么时候用：多人协作项目；希望把“提需求 -> 改代码 -> 审查 -> 合并”链路做成可追溯闭环。

来源：modelcontextprotocol/servers

2) 浏览器调试与交互：Chrome DevTools MCP

  • 能做什么：页面快照、元素交互、Console/Network 检查、Lighthouse 评估、性能 trace 采集与分析。
  • 什么时候用：线上问题复现、性能定位、可视化取证；尤其适合“只看代码看不出问题”的前端故障。

3) 浏览器自动化：Puppeteer MCP

  • 能做什么：浏览器导航、元素操作、脚本执行、截图抓取、表单流程自动化。
  • 什么时候用：重复性网页流程（采集、截图、回归）要稳定复现时。

4) 支付与业务：Stripe MCP

  • 能做什么：客户、产品、价格、支付链接、发票、订阅、退款、争议处理、余额查询；还可做 Stripe 集成问答与文档检索。
  • 什么时候用：支付业务链路需要规范化、自动化，且要降低手工后台操作风险时。

5) 数据库自动化：Neon MCP

  • 能做什么：项目/分支管理、SQL 执行、迁移准备与完成、慢查询与调优、Schema Diff、Auth/Data API、连接串管理。
  • 什么时候用：数据库改动要“先分支验证再合并主分支”时，最能体现它的价值。

6) 安全治理：Semgrep MCP

  • 能做什么：SAST/SCA 扫描、自定义规则检测、供应链扫描、平台历史发现读取（已上传结果）。
  • 什么时候用：改依赖、上线前、合规复核阶段；把安全检查纳入固定流水线。

7) 文档与框架学习：Context7 MCP

  • 能做什么：先解析库 ID，再检索官方文档和示例，拿到更可执行的用法。
  • 什么时候用：已确定技术栈，但实现细节不确定，想减少试错成本时。

8) 设计与素材链路：Figma MCP（Framelink）

  • 能做什么：读取 Figma 文件/节点结构、导出节点资源（PNG/SVG）、用于设计到开发的信息对齐。
  • 什么时候用：设计稿需要批量导出、自动切图、结构化交付给开发时。

9) 视觉生成：Replicate MCP

  • 能做什么：文生图、批量变体、预测任务查询、SVG 生成。
  • 什么时候用：海报封面、活动图、插画探索等需要多方案快速打样时。

10) Web 信息获取：Web Search MCP（Exa）

  • 能做什么：按主题检索全网内容，返回可读整理结果，支持快速信息扫描。
  • 什么时候用：需要先快速摸清领域背景，再决定深入方向时。

11) URL 内容抓取：Webfetch MCP

  • 能做什么：抓取指定 URL，并转换为 Markdown/Text/HTML 便于整理和引用。
  • 什么时候用：已知目标链接，要做归档、摘要、引用；它是抓取与转换，不是浏览器渲染执行。

🧩 内置工具（非 MCP，但同样重要）

这些不是 MCP Server，但在日常任务里同样是核心能力：

1) 代码结构化搜索与替换

  • AST Grep 工具：可做语法级别的搜索与替换，适合批量重构、统一 API 用法或跨文件改名时保证准确性。

2) 代码智能（LSP）

  • LSP 工具：跳转定义、查找引用、符号检索、诊断与重命名，适合在复杂代码里快速定位影响范围与风险。

3) 会话与历史检索

  • Session 工具：会话列表、历史读取与全文检索，适合回溯上下文与复用已有结论。

4) 后台任务控制

  • Background 工具：获取后台任务输出与取消任务，适合并行执行和长任务管理。

5) 视觉/文档解析

  • Look_at 工具：图片与 PDF 视觉解析，适合提取图表信息、识别结构化内容。

6) 交互式终端

  • Interactive Bash：面向需要持续交互的 TUI（如 tmux/vim），适合复杂命令行操作。

7) 任务清单管理

  • Todo 工具：多步骤任务拆解与进度跟踪，适合复杂项目的过程管理与交付对齐。

来源：neondatabase/neon

✅ 我当前的落地原则

为避免“越自动化越混乱”，我固定了三条规则：

  1. 先验证后执行：发布、迁移、删除都要先检查
  2. 强约束可追溯：关键动作必须有日志与结果状态
  3. 保留人工确认闸门：自动化不等于无审核

再补一条我自己踩坑后的规则：

  1. 高风险动作必须有回滚路径：例如数据库迁移、批量发布、支付操作。

📝 给未来自己的操作清单

  • 每月更新一次 Skills/MCP 清单
  • 新增能力先试运行 3 次，再决定是否固化
  • 对外输出时，把“做法 + 约束 + 失败恢复”一起写清楚

如果你也在做个人 AI 工作台，建议从一个高频场景开始（例如“内容发布”或“代码协作”），先跑通，再扩展。
