Discord机器人更新记录

Discord 机器人更新纪要：从单平台到 QQ 双端联动

这次更新的核心目标很明确：在尽量不重写既有业务逻辑的前提下，把原本运行在 Discord 的机器人能力扩展到 QQ（NapCat + OneBot v11），并解决接入过程中出现的稳定性与鉴权问题。

这次更新了什么

1) 新增 QQ 桥接入口（最小侵入）

  • 新增 qq_bridge.py，作为 OneBot v11 HTTP 上报接收端。
  • NapCat 负责接收 QQ 消息并上报给桥接，桥接再调用 NapCat HTTP API 回发消息。
  • 业务上复用了原有主链路：_call_ai -> _parse_command_intent -> _execute_command_from_intent。

这意味着：原本 Discord 端的 AI 对话、命令意图解析、命令执行能力，基本可以直接在 QQ 端复用。

2) 补齐对接配置与启动流程

  • 在 config/config.yaml 中增加 QQ 桥接相关配置项（QQ_BRIDGE_*、NAPCAT_API_ROOT、NAPCAT_ACCESS_TOKEN、ONEBOT_* 等）。
  • 在 README.md 新增完整的 QQ 对接说明，覆盖网络配置、鉴权、安全建议和已知能力边界。
  • 新增 start_qq_robot.bat，用于一键启动当前目录下的 QQ 桥接进程（不强依赖自动拉起 NapCat）。

3) 修复了多轮线上问题（这部分最关键）

接入过程中踩到的坑主要来自 OneBot 鉴权和回调状态码处理，本次已逐一修复：

  • 修复入站鉴权兼容性：支持多种 token 传递格式（Bearer、Token、原始值、X-Access-Token、query）。
  • 增加 loopback 放行策略：本机来源请求在缺少 token 时可放行，避免本地部署被 401 卡死。
  • 修复出站调用鉴权：桥接请求 NapCat API 时同时兼容 Header 与 query token 方式。
  • 修复 URL 隐性错误：清理 NapCat 上报地址尾部空格导致的不稳定行为。
  • 修复 retcode 误判：retcode=0 不再被错误识别为失败。
  • 修复回调噪音：即使内部处理异常，事件回调也返回 200，避免 NapCat 持续报 Unexpected status code: 500。
  • 增强日志可观测性：鉴权失败日志增加脱敏字段，方便快速定位头部/参数不匹配问题。

4) 当前能力边界

  • QQ 端已稳定支持：文本对话、自动意图命令执行、图片消息回传。
  • 暂未完全对齐：Discord 按钮/模态等交互式 UI（QQ 不同协议形态，不能 1:1 复刻）。
  • 非图片文件在 QQ 桥接侧仍有能力边界，需要后续按 OneBot 文件上传接口单独扩展。

架构变化（简版）

为什么这次改造值得做

这次不是简单“多加一个平台”，而是把机器人从“单入口应用”升级成“多入口能力内核”：

  • 平台层（Discord / QQ）和业务层（AI、命令、记忆、任务）开始解耦。
  • 后续再接入新平台时，不需要重写整套能力，只需补新适配层。
  • 线上问题排查效率明显提升（鉴权、回调、发送链路都有可观测日志）。

下一步计划

  1. 把更多 Discord 专属交互迁移为跨平台可复用流程（优先文本交互）。
  2. 补齐 QQ 非图片文件上传能力（对接 OneBot 文件相关接口）。
  3. 进一步拆分 bot.py，把平台无关逻辑沉淀到 services/，降低维护成本。

如果你也在做“从 Discord 到 QQ”的迁移，这次实践的核心经验是：先保住业务核心，再做协议适配；先把链路跑通，再逐步补齐体验。
